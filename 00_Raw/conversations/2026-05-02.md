## 🖼️ Admin 페이지 최종 시각적 가이드라인 (Glassmorphism & Rose Tone)

Admin 페이지는 '신뢰감'과 '친근함(Rose Tone)'을 기반으로 하면서도 '데이터의 명확성(Clarity)'을 최우선으로 확보하는 것이 핵심 목표입니다. Glassmorphism 기법을 적용하여 깊이감을 주되, 정보 접근성은 극대화해야 합니다.

### 1. 색상 팔레트 (Rose Tone System)

브랜드 정체성인 '로즈 톤'을 중심으로 따뜻함과 전문성을 동시에 확보하는 컬러 시스템입니다.

| 요소 | 역할 | HEX 코드 | 설명 |
| :--- | :--- | :--- | :--- |
| **Primary Rose** | 메인 액센트/CTA (버튼, 활성 상태) | `#E07A5F` | 따뜻하고 친근한 핵심 색상. 사용자의 행동을 유도합니다. |
| **Secondary Rose** | 보조 강조/서브 컬러 | `#F4ACB7` | 배경이나 카드에 부드러운 톤으로 적용하여 깊이를 더합니다. |
| **Glass Background (Light)** | Glassmorphism 기본 배경 | `rgba(255, 255, 255, 0.3)` | 투명도를 활용한 유리 효과의 베이스. |
| **Glass Border/Surface** | 카드 및 패널 경계선 | `rgba(255, 255, 255, 0.15)` | 미묘한 빛 반사와 깊이감을 부여합니다. |
| **Text (Primary)** | 본문 및 핵심 데이터 | `#333333` | 높은 가독성을 위한 진한 회색(Black 대신 사용). |
| **Text (Secondary)** | 보조 정보 및 레이블 | `#6B6B6B` | 부드러운 톤으로 보조 정보를 처리합니다. |

### 2. 타이포그래피 시스템 (Serif & Clarity)

전문성과 친근함을 동시에 전달하기 위해 Serif 계열의 타이포그래피를 사용하며, 데이터 가독성을 확보하는 것이 중요합니다.

*   **Font Family:** **Serif 계열** (예: Playfair Display, Merriweather 또는 유사한 세리프 폰트)
    *   *적용 위치:* 헤더, 섹션 제목, 핵심 데이터 라벨 등 시각적 강조가 필요한 곳에 적용하여 브랜드의 따뜻함을 표현합니다.
*   **Font Family (Body/Data):** **Sans-serif 계열** (예: Inter, Roboto 또는 유사한 산세리프 폰트)
    *   *적용 위치:* 실제 데이터 표(Table), 상세 설정 항목 등 정보 전달이 주 목적인 곳에 적용하여 명확성을 확보합니다.
*   **Hierarchy:**
    *   **H1/H2 (섹션 제목):** Serif 계열, 볼드 처리, 충분한 자간(Letter Spacing) 적용. (브랜드 톤 강조)
    *   **Label/Key:** Sans-serif, 적절한 크기. (명확성 확보)
    *   **Data/Value:** Sans-serif, 높은 대비로 명확하게 표시.

### 3. Glassmorphism 레이아웃 적용 가이드

Admin 페이지의 각 패널(Sidebar, Main Content, Data Cards)에 다음 규칙을 적용합니다.

1.  **배경 처리 (The Glass Effect):**
    *   모든 컨테이너는 반투명한 배경(`rgba(255, 255, 255, 0.3)`)을 사용하며, 경계선은 매우 얇고 희미하게 설정합니다 (`border: 1px solid rgba(255, 255, 255, 0.15);`).
    *   **팁:** 배경색이 너무 어두우면 Glassmorphism 효과가 약해지므로, 메인 배경은 미드톤의 밝은 로즈 톤(`F9F8F7` 등)을 사용하여 전체적인 따뜻함을 유지해야 합니다.

2.  **깊이감 (Depth & Shadow):**
    *   **Soft Shadow:** 부드러운 그림자(Box-shadow)를 사용하여 화면에서 살짝 떠 있는 듯한 입체감을 부여합니다.
        *   `box-shadow: 0 4px 15px rgba(0, 0, 0, 0.08);` (매우 은은하고 소프트하게 설정)

3.  **레이아웃 구조:**
    *   **Sidebar/Navigation:** 상대적으로 불투명도를 높여 중요한 내비게이션 요소로 기능하게 합니다. (Glassmorphism의 적용 강도 차등화)
    *   **Data Panels (Cards):** 모든 데이터는 명확한 경계를 가진 카드 형태로 분리하여, 한눈에 정보 구분을 용이하게 합니다. 각 카드는 Secondary Rose 색상을 미묘하게 배경으로 활용할 수 있습니다.

## 📊 핵심 KPI 및 퍼널 분석의 수익화 연관성 전략 검토

Admin 페이지에 포함될 핵심 KPI 및 퍼널 분석 기능은 단순히 데이터를 보여주는 것을 넘어, **수익화 모델의 효율성을 측정하고 다음 가격/번들 전략을 결정하는 데 직접적인 근거**가 되어야 합니다. 현재 정의된 목표(수익화 모델 1개 가설 검증)를 달성하기 위해 가장 중요한 지표와 그 연관성을 다음과 같이 재확인하고 우선순위를 설정합니다.

### 1. 핵심 KPI 및 수익화 연결 분석

| KPI/분석 항목 | 측정 목적 (What to Measure) | 수익화와의 직접적 연관성 (Why it Matters for Revenue) | 우선순위 |
| :--- | :--- | :--- | :--- |
| **A. 진단 -> 가입 전환율** | 사용자가 무료 진단 후 유료 플랜(프리미엄 리포트 등)으로 전환하는 비율 측정. | **가장 중요.** 사용자 확보 비용(CAC) 대비 실제 매출 발생 효율을 측정하여 마케팅 예산 배분과 가격 민감도를 파악. | **1순위 (Critical)** |
| **B. LTV (고객 생애 가치)** | 한 사용자가 플랫폼에 머무는 기간 동안 창출하는 총 예상 수익. | 장기적인 구독 모델(Subscription)의 성공 여부를 판단. 리텐션 전략 및 가격 인상 가능성을 결정. | **1순위 (Critical)** |
| **C. 리포트별 평균 매출 (ARPU/ARPPU)** | 각 프리미엄 리포트(번들 옵션)에 대한 사용자당 평균 지불 금액 측정. | 수익화 모델(가격 전략)의 성공 여부 검증. 어떤 번들이 가장 높은 가치를 제공하는지 확인하여 제품 포트폴리오를 조정. | **2순위 (High)** |
| **D. 제휴 네트워크 성과** | 제휴 미용실 연계를 통한 실제 예약/리포트 구매 발생 건수 및 수익 기여도. | 제휴 파트너십의 가치 평가 및 커미션 구조 최적화. B2B 또는 제휴 수익 모델의 실효성 검증. | **3순위 (Strategic)** |
| **E. 이탈률 (Churn Rate)** | 구독 해지 또는 서비스 이용 중단을 보이는 비율 측정. | 고객 유지 전략(Retention)의 핵심 지표. 서비스 품질과 가격 책정 간의 균형점을 찾음. | **2순위 (High)** |

### 2. 최종 우선순위 재확인 및 권고

수익화 목표가 '수익화 모델 검증'에 있다면, **'사용자가 돈을 지불하는 단계'와 '그들이 얼마나 오래 머무르는지'**를 측정하는 것이 최우선입니다.

1.  **🥇 1순위: 전환율 (A)** - 무료 사용자로부터 유료 사용자로의 변환 경로를 명확히 하여 마케팅 효율성을 즉시 개선합니다.
2.  **🥈 2순위: LTV 및 이탈률 (B, E)** - 확보된 고객이 장기적으로 수익을 창출하는지 확인하여 가격 전략과 서비스 품질에 대한 근거를 마련합니다.
3.  **🥉 3순위: ARPU/번들별 매출 (C)** - 현재 설정한 가격 전략(번들 옵션) 중 어떤 것이 시장에서 가장 매력적인지 데이터 기반으로 검증합니다.

## 1. 데이터 유출의 비즈니스 임팩트 분석

데이터 유출은 단순한 기술적 사고를 넘어, 미드미의 비즈니스 모델 자체를 붕괴시적 위험을 내포합니다.

### 📉 브랜드 신뢰도(Brand Trust) 측면
- **브랜드 정체성 훼손:** '친절하고 전문적인' 브랜드 톤이 '무책임하고 위험한' 이미지로 즉각 전환.
- **바이럴 역효과:** AI 서비스 특성상 SNS 공유가 핵심 마케팅인데, 보안 사고 발생 시 부정적 바이럴이 신규 고객 획득 비용(CAC)을 기하급수적으로 상승시킴.

### 📉 LTV(고객 생애 가치) 및 재무적 측면
- **Churn Rate(이탈률) 급증:** 민감 정보(얼굴 사진) 유출 시, 기존 사용자의 재방문율이 0에 수렴하며 LTV가 급락함.
- **LTV 하락 메커니즘:** `LTV = (평균 결제 금액 × 구매 빈도) × 고객 유지 기간`. 유출 사고는 '고객 유지 기간'을 강제로 종료시킴.
- **직접적 비용 발생:** PIPA(개인정보보호법) 위반에 따른 과징금(매출액의 일정 비율), 집단 소송 비용, 시스템 복구 및 보안 감사 비용 발생.

| :--- | :--- | :--- | :--- | :--- | :--- |

# 🛡️ 미드미(midmi) 보안 아키텍처 설계서

## 1. 인증 및 인가 레이어 (Authentication & Authorization)
**목표:** NextAuth.js를 활용한 세션 관리 및 RBAC(역할 기반 접근 제어) 구현

*   **NextAuth.js + JWT 전략:**
    *   **Session Strategy:** `jwt` 전략 사용. 세션 데이터에 `role` (USER, ADMIN) 및 `userId`를 포함하여 클라이언트/서버 모두에서 즉각적인 권한 확인 가능하도록 설계.
    *   **Token Security:** JWT 서명 시 `AUTH_SECRET`을 사용하여 변조 방지. `HttpOnly`, `Secure`, `SameSite=Lax` 쿠키 설정을 통해 XSS 및 CSRF 공격 차단.
*   **RBAC (Role-Based Access Control):**
    *   **Middleware 활용:** Next.js Middleware에서 `/admin/*` 경로에 대해 `role === 'ADMIN'` 체크 로직 강제 적용.
    *   **API Route Guard:** 모든 API 엔드포인트 상단에 `getServerSession`을 통한 세션 유효성 및 권한 검증 로직 표준화.

## 2. 데이터 보호 레이어 (Data Encryption & Integrity)
**목표:** Level 3(민감정보) 데이터의 물리적/논리적 암호화

*   **At-Rest Encryption (저장 시 암호화):**
    *   **AES-256-GCM 적용:** 생체 특징점 및 개인 식별 데이터는 Node.js `crypto` 모듈의 `aes-256-gcm` 알고리즘을 사용하여 암호화. (GCM 모드를 사용하여 데이터 무결성(Integrity)까지 검증)
    *   **Key Management:** 암호화 키는 코드에 포함하지 않으며, AWS Secrets Manager 또는 Vercel Environment Variables를 통해 주입.
*   **In-Transit Encryption (전송 시 암호화):**
    *   **TLS 1.3 강제:** 모든 통신은 HTTPS를 통해서만 이루어지도록 HSTS(HTTP Strict Transport Security) 설정.
*   **Input Validation (Injection 방지):**
    *   **Zod 스키마 검증:** 모든 API 요청 Body, Query, Params는 `Zod`를 통해 타입 및 패턴 검사. (SQL Injection 및 NoSQL Injection 원천 차단)

## 3. API 보안 및 트래픽 제어 (API Security & Rate Limiting)
**목표:** DoS 공격 방지 및 서비스 가용성 확보

*   **API Rate Limiting:**
    *   **Upstash Redis 기반 구현:** Vercel Edge Middleware에서 Upstash(Redis)를 사용하여 IP 및 User ID 기반의 요청 횟수 제한.
    *   **Tier별 제한:** 
        *   `Public API`: 초당 10회 제한.
        *   `Authenticated API`: 초당 50회 제한.
        *   `Admin API`: 초당 5회 제한 (보안 극대화).
*   **CORS Policy:** `midmi.com` 도메인 외의 모든 외부 호출을 차단하도록 화이트리스트 기반 설정.

## 4. 스토리지 보안 (S3 Security Architecture)
**목표:** 사용자 얼굴 원본 이미지(Level 3)의 비인가 접근 차단

*   **Private Bucket 구성:** S3 Bucket의 `Public Access`를 완전히 차단하고, 모든 객체에 대해 `Block Public Access` 설정.
*   **Presigned URL 패턴:**
    *   이미지 조회 시 클라이언트가 S3에 직접 접근하는 것이 아니라, 서버(Next.js API)가 **유효 기간이 극히 짧은(예: 60초) Presigned URL**을 생성하여 전달.
    *   이를 통해 URL 탈취 시에도 재사용 불가능하게 설계.
*   **IAM Least Privilege:** Next.js 서버가 사용하는 IAM Role에는 해당 Bucket에 대한 `s3:PutObject`, `s3:GetObject` 권한만 부여 (전체 권한 부여 금지).

## 5. 인프라 및 운영 보안 (Infrastructure & DevOps)
**목표:** 환경 변수 관리 및 시스템 가시성 확보

*   **Secret Management:** `.env` 파일은 로컬 개발용으로만 사용하며, 운영 환경에서는 Vercel/AWS의 보안 저장소를 사용.
*   **Logging & Monitoring:** 
    *   **Audit Log:** 데이터 수정/삭제/권한 변경 등 주요 액션은 별도의 `audit_logs` 테이블에 기록 (누가, 언제, 무엇을).
    *   **Error Tracking:** Sentry를 연동하여 보안 관련 에러(Unauthorized, Forbidden) 발생 시 즉시 알림.

## 2. Backend: 데이터 보안 및 로그 관리

### 2.1 AES-256-GCM 암호화 적용
개인정보(이름, 연락처, 이메일 등) 및 민감한 진단 결과 데이터는 반드시 암호화하여 저장한다.
- **알고리즘:** `AES-256-GCM` (Authenticated Encryption) 사용. (데이터 무결성 검증을 위해 CBC 대신 GCM 권장)
- **키 관리 (Key Management):**
  - 암호화 키는 소스 코드에 포함하지 않으며, `AWS KMS` 또는 `HashiCorp Vault`를 통해 관리한다.
  - 환경 변수(`process.env.ENCRYPTION_KEY`)를 통해 주입하되, 주기적(연 1회) 키 로테이션을 수행한다.
- **구현 로직:**
  - **Encryption:** `Plaintext + Key + IV(Initialization Vector) -> Ciphertext + Auth Tag`
  - **Decryption:** `Ciphertext + Auth Tag + IV + Key -> Plaintext` (태그 검증 실패 시 즉시 에러 처리)

### 2.2 접속 및 감사 로그(Audit Log) 관리
모든 데이터 접근 및 변경 이력은 추적 가능해야 한다.
- **로그 기록 대상:** 사용자 인증(Login/Logout), 권한 변경(RBAC), 개인정보 조회/수정/삭제, API 호출 이력.
- **로그 구조 (JSON Format):**
  ```json
  {
    "timestamp": "ISO8601",
    "actor_id": "user_uuid",
    "action": "DATA_ACCESS | DATA_UPDATE",
    "resource": "user_profile",
    "ip_address": "xxx.xxx.xxx.xxx",
    "user_agent": "...",
    "status": "SUCCESS | FAILURE"
  }
  ```
- **보관 주기 및 계층화 저장:**
  - **Hot Storage (최근 3개월):** `Elasticsearch` 또는 `CloudWatch Logs`에 저장하여 즉시 검색 가능하도록 유지.
  - **Cold Storage (3개월 ~ 2년):** `Amazon S3` (Glacier Instant Retrieval)로 아카이빙. `S3 Lifecycle Policy`를 통해 2년 경과 시 자동 삭제 설정.

### 2.3 데이터 파기 프로세스 (Data Destruction)
사용자 탈퇴 또는 법적 보관 기간 만료 시, 복구가 불가능한 상태로 삭제한다.
- **논리적 삭제(Soft Delete) 후 물리적 삭제(Hard Delete):**
  - 탈퇴 즉시 `is_deleted: true` 처리하여 서비스 노출 차단.
  - 배치 작업(Cron Job)을 통해 30일 경과 후 DB에서 `DELETE` 쿼리 실행.
- **Crypto-shredding (암호화 키 파기):**
  - 특정 사용자의 데이터 전용 키를 별도 관리할 경우, 해당 사용자의 암호화 키를 삭제함으로써 데이터의 복구 가능성을 원천 차단한다.
- **백업 데이터 관리:** 백업본 내 개인정보 역시 동일한 파기 주기를 따르도록 백업 스냅샷 삭제 정책을 동기화한다.
