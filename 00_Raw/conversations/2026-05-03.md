데이터는 LTV 측정과 보안 요구사항(RBAC)을 중심으로 설계하며, 모든 민감 정보는 암호화되어 저장되어야 합니다.

#### 2.1. User Schema (사용자 및 권한 관리)
**목표:** 사용자 인증 및 역할 기반 접근 제어(RBAC)의 근간.
*   `user_id` (UUID/BigInt, Primary Key)
*   `email` (String, Unique, Encrypted by AES-256)
*   `password_hash` (String, Argon2 또는 bcrypt 해시 적용)
*   `role` (Enum: `ADMIN`, `MANAGER`, `USER`) - **RBAC 핵심 필드**

#### 2.2. Report Schema (AI 진단 및 리포트 데이터)
**목표:** AI 분석 결과와 사용자 데이터를 연결하여 신뢰도를 높임.
*   `report_id` (UUID, Primary Key)
*   `user_id` (Foreign Key to User)
*   `analysis_date` (Date)
*   `style_recommendation` (JSONB/Text: AI가 제시한 스타일 상세 데이터)
*   `trust_score` (Float: 데이터 기반 신뢰도 점수, 0.0 ~ 1.0) - **Glassmorphism 시각화의 핵심 지표.**
*   `data_source_hash` (String: 원본 데이터 무결성 검증용 해시값)

#### 2.3. Transaction Schema (구독 및 결제 기록)
**목표:** LTV 측정을 위한 재정적 흐름 추적.
*   `transaction_id` (UUID, Primary Key)
*   `user_id` (Foreign Key to User)
*   `type` (Enum: `SUBSCRIPTION`, `ADJUSTMENT`, `REFUND`)
*   `amount` (Float/Decimal)
*   `status` (Enum: `SUCCESS`, `PENDING`, `FAILED`)
*   `transaction_date` (Timestamp)

### 4. 보안 및 데이터 흐름 명세 (Security & Data Flow Specification)

**🎯 목표:** AES-256 암호화와 RBAC를 서비스 전체에 통합.

1.  **데이터 저장 시점 (Persistence):**
    *   사용자 정보(`email`, 민감한 설정)는 DB에 저장 전 **AES-256-GCM** 알고리즘을 사용하여 서버 측에서 암호화하여 저장해야 합니다.
    *   민감 데이터의 키(Key)는 환경 변수(Environment Variables)를 통해 안전하게 관리하며, 절대 코드 내에 하드코딩하지 않습니다.

2.  **API 접근 제어 (RBAC Implementation):**
    *   모든 백엔드 API 엔드포인트(`POST`, `GET` 등) 앞단에서 JWT 검증 및 사용자 역할(`role`) 확인 로직을 필수적으로 삽입합니다.
    *   예: `/api/reports/create` 요청 시, 인증된 사용자가 `MANAGER` 또는 `ADMIN` 역할이어야만 접근 허용.

3.  **데이터 파기 정책 (Data Deletion Policy):**
    *   **Soft Delete:** 사용자 및 트랜잭션 데이터는 즉시 삭제하지 않고, `is_deleted: true` 플래그를 설정하여 논리적 삭제를 수행합니다. 이는 LTV 추적을 위해 필요합니다.
    *   **Hard Delete (복구 불가능한 삭제):** 사용자가 명시적으로 요청하거나, 법적 요구사항 발생 시에만 실제 데이터베이스 레코드에서 암호화 키와 함께 데이터를 **복구 불가능하게 파기(Cryptographic Shredding)**하는 프로세스를 백엔드 레이어에서 구현합니다.

# 📈 핵심 KPI 통합 및 LTV 기반 데이터 흐름 전략 프레임워크

본 전략은 웹사이트 구조 내에 **구독 전환율(Conversion Rate)**과 **고객 생애 가치(LTV)**를 측정하고 최적화하기 위한 데이터 흐름의 전략적 프레임을 제시합니다. 모든 데이터는 '데이터 기반 신뢰'라는 브랜드 핵심 가치와 일치하도록 설계되어야 합니다.

## 1. 핵심 KPI 정의 및 목표 설정 (The Goal)

| KPI | 정의 | 측정 목표 | 비즈니스 연관성 |
| :--- | :--- | :--- | :--- |
| **전환율 (CR)** | 무료 사용자 $\rightarrow$ 유료 구독 전환 비율 | **최소 5% 이상** 유지 (A/B 테스트 기반) | 직접적인 매출 기여도 측정. |
| **LTV (Lifetime Value)** | 한 사용자가 플랫폼에 기여하는 총 예상 수익 | **구독 유지율(Retention Rate) 극대화** | 장기적인 수익성과 고객 확보 비용(CAC) 효율성 측정. |
| **데이터 신뢰 점수 (Trust Score)** | 사용자 피드백, 데이터 정확도 기반의 주관적/객관적 평가 | **평균 4.5점 이상** 유지 (Glassmorphism 시각화 목표) | 프리미엄 가격 책정의 근거 및 이탈 방지 요인. |

## 2. 데이터 흐름 설계 (The Flow)

데이터는 사용자 여정(User Journey)에 따라 세 단계로 구분하여 수집하고 통합해야 합니다.

### 단계 1: 인입/진단 단계 (Acquisition & Diagnosis - **Trust Building**)
**목표:** 잠재 고객의 니즈를 파악하고 신뢰도를 극대화하여 다음 단계로 유도합니다.

*   **수집 데이터:** 스타일 진단 결과, 사용한 AI 인터랙션 로그, 리포트 열람 시간, 가입 시점 및 경로.
*   **LTV 연관 지표:** **데이터 신뢰 점수(Trust Score)**의 초기 측정 항목으로 활용. (예: "당신의 데이터는 95% 정확합니다.")
*   **통합 전략:** 진단 결과 페이지에 '데이터 기반 신뢰 시각화' 요소를 Glassmorphism 형태로 배치하여 즉각적인 신뢰감을 제공하고, 다음 단계(결제)로의 심리적 장벽을 낮춥니다.

### 단계 2: 전환 단계 (Conversion - **Monetization**)
**목표:** 사용자가 프리미엄 구독으로 전환하도록 유도합니다.

*   **수집 데이터:** 가격 페이지 방문 로그, 옵션 비교 기록, 결제 시도 및 실패 로그, 구독 결정 시점의 행동 패턴.
*   **LTV 연관 지표:** **구독 전환율(CR)** 측정의 핵심 구간. 이탈 예측 모델을 위한 초기 변수로 사용됩니다.
*   **통합 전략:** 가격 책정 구조는 LTV 지표에 기반하여 Tiered Pricing으로 설정합니다. 프리미엄 티어는 단순 기능 제공이 아닌, '더 높은 데이터 신뢰도'와 'AI 전문가 접근 권한'이라는 가치를 명확히 하여 전환의 동인으로 만듭니다.

### 단계 3: 유지/활용 단계 (Retention & Value - **LTV Maximization**)
**목표:** 구독자에게 지속적인 가치를 제공하여 이탈을 방지하고 LTV를 극대화합니다.

*   **수집 데이터:** 월별 리포트 재방문율, AI 기능 사용 빈도, 피드백(CS 채널), 서비스 이용 패턴 변화.
*   **LTV 연관 지표:** **구독 유지율(Retention Rate)** 및 **이탈 예측 지표(Churn Prediction)** 산출.
*   **통합 전략:** 사용자 대시보드는 단순히 데이터 표시를 넘어, "당신의 스타일 변화 추이"와 같은 개인화된 피드백을 제공하여 지속적인 사용 동기를 부여합니다.

## 3. 웹사이트 구조 내 KPI 통합 방안 (Integration Map)

| 웹사이트 영역 | 측정할 핵심 지표 | LTV/CR 연관성 | 데이터 시각화 전략 |
| :--- | :--- | :--- | :--- |
| **Hero Section** | 초기 클릭률 (CTR), 세션당 평균 체류 시간 | 잠재 고객의 관심도 측정. 전환율 예측. | 신뢰 요소(Trust Score)를 Glassmorphism으로 시각화하여 즉각적 신뢰 확보. |
| **스타일 진단 결과 페이지** | 리포트 다운로드율, 상세 분석 페이지 방문율 | 데이터 수집 성공률 측정. 다음 단계로의 이동 동기 측정. | 데이터 정확도(Trust Score)를 그라데이션으로 표현하여 전문성 강조. |

## 4. 최종 권고 (Recommendation)

**결정:** 모든 데이터 흐름은 **'데이터 신뢰 $\rightarrow$ 전환 유도 $\rightarrow$ 유지'**의 선형적이고 순환적인 구조를 따라야 합니다.

2.  **LTV 중심 설계:** 기능 개발 시, 단순히 기능을 나열하는 것이 아니라 각 기능 사용이 **장기적인 가치(LTV)**에 어떻게 기여하는지 정의하고, 이를 대시보드에 명확히 연결해야 합니다.

## 🎯 Trust Score UI/UX 최종 요구사항 명세 (Designer 전달용)

본 요구사항은 **'데이터 기반 신뢰 구축이 구독 전환의 핵심 동인'**이라는 비즈니스 목표를 시각적으로 증명하는 데 중점을 둡니다. 'Trust Score' 컴포넌트는 단순한 지표가 아니라, 사용자가 더 높은 가치(LTV)를 지불할 만한 근거를 제공해야 합니다.

### 1. 핵심 연결 논리 (Business Logic Mapping)

| 비즈니스 목표 | Trust Score의 역할 | 시각적 구현 원칙 |
| :--- | :--- | :--- |
| **구독 전환 극대화** | 사용자가 결제하기 전에 '이 서비스가 나에게 신뢰할 만한 가치를 제공한다'는 심리적 확신을 즉시 제공. | 점수가 높을수록 시각적 안정감과 깊이가 느껴지도록 디자인. (Rose Tone의 깊은 그라데이션 활용) |
| **LTV 측정/관리** | 사용자의 현재 신뢰 수준(Trust Score)이 잠재적 장기 가치(LTV)에 미치는 영향을 명확히 인지시킴. | 점수 변화에 따라 Glassmorphism 효과의 투명도나 색상의 깊이가 동적으로 변화하도록 설계하여, 데이터가 '살아있음'을 표현. |
| **신뢰 확보 (Trust Building)** | 서비스의 안전성과 전문성을 시각적 요소(Glassmorphism)를 통해 증명. | 불안정하거나 낮은 점수에서는 경고색/미묘한 어두운 톤을 사용하여 주의를 환기시키고, 높은 점수에서는 부드러운 Rose Tone 그라데이션으로 신뢰감을 극대화. |

### 2. 컴포넌트 상세 디자인 지침 (Visual Specification)

#### A. 데이터 표현 방식: 동적 깊이(Dynamic Depth) 활용

*   **Trust Score 게이지:** 단순한 막대(Bar) 형태 대신, **중심에 배치된 원형 또는 다이아몬드 형태의 심볼**을 중심으로 점수가 채워지는 방식으로 디자인합니다.
*   **Glassmorphism 적용:** 이 심볼 컨테이너 전체에 `backdrop-filter: blur(16px);`를 적용하여 배경과의 분리감을 극대화하고, 데이터 레이어임을 시각적으로 강조합니다.

#### B. 색상 및 그라데이션 전략 (Rose Tone & Trust)

*   **낮은 신뢰도 (Low Trust):** 점수가 낮을 경우, 배경에 미묘하게 **차가운 톤의 회색(Cool Gray)**이 Glassmorphism 뒤에 살짝 비치도록 하여 잠재적 위험성을 암시합니다.
*   **중간 신뢰도 (Medium Trust):** `Primary Rose (#E07A5F)`가 주를 이루며, 부드러운 그라데이션을 통해 전문성과 친근함을 동시에 전달합니다.
*   **높은 신뢰도 (High Trust):** 가장 깊고 풍부한 **Rose Tone의 따뜻한 그라데이션**을 사용하여 최고 수준의 안정감과 프리미엄 가치를 시각적으로 표현합니다.

#### C. 타이포그래피 및 레이블

*   **스코어 텍스트:** Serif 계열(예: Playfair Display)을 사용하여 전문성과 클래식함을 유지합니다.
*   **보조 설명 (Contextual Label):** 점수 아래에 "LTV 기반 신뢰 지표"와 같은 문구를 Glassmorphism의 투명한 배경 위에 작게 배치하여, 시각적 요소가 아닌 **데이터의 근거**를 함께 제시합니다.

### 3. 실행 우선순위 (Action Priority)

1.  **최우선:** LTV에 따른 색상/그라데이션 변화 규칙을 정의하고 Mockup에 적용한다.
2.  **차선:** Glassmorphism 블러(Blur) 효과가 Rose Tone의 깊이를 충분히 표현하도록 16px 설정을 유지하며 테스트한다.
